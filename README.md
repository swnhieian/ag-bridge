# AG Bridge

Language: [English](README.md) | [简体中文](README.zh-CN.md)

`AG Bridge` is a VS Code / Antigravity extension that starts a lightweight local bridge service and exposes Antigravity Language Server through stable HTTP and SSE interfaces for automation, session takeover, frontend integration, and scripting.

## Overview

- Starts a local bridge service inside the extension host
- Provides a dashboard for the listening address, port, data directory, and sessions
- Lets you copy the current bridge base URL
- Syncs existing Antigravity sessions and lets you continue sending messages to them
- Re-syncs already attached live sessions from history so missed turns can be backfilled after temporary reactive stream gaps
- Streams text output, thinking, command output, approval requests, and full step details through HTTP / SSE
- Supports exporting full session data
- Supports CLI, Python SDK, and external script integration
- Includes auto-approval controls for semi-automated or fully automated workflows
- Keeps auto-approval active even when the bridge falls back from reactive streaming to polling

## Typical Use Cases

- Read AG session streams from your own frontend or backend
- Send messages to AG directly from scripts instead of driving the desktop UI
- Pipe AG output into tests, agents, workflow orchestration, or other tooling
- Access full step data instead of only the final text output

## Requirements

Recommended environment:

- macOS
- Antigravity.app installed
- Logged into Antigravity at least once
- Node.js 22+

The default mode is `connect`, which attaches to an already running Antigravity Language Server.

## Installation

If the extension is published on `Open VSX`, installing it from the Antigravity / VS Code extension marketplace is recommended.

You can also install it from a VSIX package.

1. Open the Extensions view.
2. Click the `...` menu.
3. Select `Install from VSIX...`.
4. Pick `ag-bridge-0.1.3.vsix`.

Command line installation:

```bash
code --install-extension ag-bridge-0.1.3.vsix --force
```

After installation, running `Developer: Reload Window` once is recommended.

## Getting Started

The extension starts the local bridge service automatically by default.

Default configuration:

- Host: `127.0.0.1`
- Port: `9464`

If the default port is occupied, the extension automatically switches to the next available port.

You can open the dashboard in these ways:

- `AG Bridge: 查看状态`
- `AG Bridge: 打开状态面板`
- Click the `AG Bridge` status bar item

The dashboard shows:

- Listening address
- Requested port and actual port
- Data directory
- Current workspace
- Session statistics
- Session list
- Auto-approval settings

## Common API Examples

Check service status:

```bash
curl http://127.0.0.1:9464/status
```

Create a session:

```bash
curl -X POST http://127.0.0.1:9464/sessions \
  -H 'Content-Type: application/json' \
  -d '{
    "mode": "connect",
    "workspacePath": "/absolute/path/to/project",
    "model": "claude-4-sonnet"
  }'
```

Send a message:

```bash
curl -X POST http://127.0.0.1:9464/sessions/<sessionId>/messages \
  -H 'Content-Type: application/json' \
  -d '{
    "text": "Please reply with OK only.",
    "model": "auto"
  }'
```

Subscribe to the event stream:

```bash
curl -N http://127.0.0.1:9464/sessions/<sessionId>/stream
```

Sync existing AG sessions:

```bash
curl -X POST http://127.0.0.1:9464/ag-sessions/sync \
  -H 'Content-Type: application/json' \
  -d '{}'
```

`POST /ag-sessions/sync` is safe to call periodically. Besides attaching newly discovered AG sessions, it also re-runs history reconciliation for already attached live sessions, which helps backfill turns that were missed during temporary stream interruptions.

## Session Identity

`session.id` is generated by the bridge itself and is used for bridge routing, persistence, and CLI / HTTP operations.

`cascadeId` is the native Antigravity session identifier returned by AG.

`trajectoryId` may appear in AG session discovery results, but it is not the bridge session id.

`POST /sessions` can now accept an optional caller-provided `sessionId`. If omitted, the bridge still generates one automatically.

Existing sessions can be resumed through `POST /sessions/:id/resume`, and the CLI also supports `resume` and `chat --last`.

For CLI flows that target a specific session id, you can also opt into `--create-if-missing` to create that session automatically when it does not exist yet.

## Step Detail Support

If you need full AG step data in a frontend, consume `cascade.step.new` and `cascade.step.updated` directly.

The `data.step` field in these events contains full step detail, including summarized fields, the current oneof payload, and protobuf-style raw JSON.

## Model Selection

CLI, HTTP, and the Python SDK all support selecting a model through the `model` field.

Common examples:

- `claude-4-sonnet`
- `google-gemini-2-5-pro`
- `sonnet`
- `pro`
- `flash`
- `auto`
- `recommended`

View the models currently returned by the language server:

```bash
node ./bin/ag-bridge.js list-models
```

When `--base-url` is omitted, the Node CLI resolves the bridge in this order: `AG_BRIDGE_URL`, workspace discovery from the local bridge registry, then `http://127.0.0.1:9464`.

Resume the most recent session:

```bash
node ./bin/ag-bridge.js resume --last
```

The repository also ships a first-class Node CLI entrypoint at `bin/ag-bridge.js`, so source builds, local scripts, and CI can reuse the same command surface without reaching into `dist/` directly.

If you want a single executable instead of running through `node`, build a native CLI binary with:

```bash
npm run package:cli:binary
```

This now generates both a raw executable and a compressed distribution archive under `artifacts/cli/`, for example:

- `artifacts/cli/ag-bridge-cli-darwin-arm64`
- `artifacts/cli/ag-bridge-cli-0.1.3-darwin-arm64.tar.gz`

To build the common release set in one go:

```bash
npm run package:cli:binary:all
```

The script supports `darwin-x64`, `darwin-arm64`, `linux-x64`, `linux-arm64`, `win32-x64`, and `win32-arm64` through `--target` or `--targets`.

`package:cli:binary` defaults to the current host target, while `package:cli:binary:all` builds a mainstream multi-platform set for release use.
