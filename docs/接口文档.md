# 接口文档

## 总览

`AG Bridge` 现在提供三层入口：

- VS Code 扩展命令
- 本地 HTTP / SSE 接口
- Python SDK / CLI

默认地址仍然是：

- `http://127.0.0.1:9464`

但如果默认端口被占用，服务会自动顺延到后续端口，最终实际地址以 `GET /status` 返回值为准。

如果是前端直接对接流式事件，推荐优先看本文的 `GET /sessions/:id/stream` 和“事件类型”两节；step 详情现在会完整落在 `cascade.step.new` / `cascade.step.updated` 的 `data.step` 里。

时间语义补充：

- `timestamp`
  - 对外主时间
  - 优先使用 AG 内部 protobuf metadata 里记录的真实事件时间
  - 如果该类事件没有 AG 侧时间，才回退为 bridge 本地记录时间
- `recordedAt`
  - bridge 实际写入这条事件的时间
- `sourceTimestamp`
  - 可选
  - 当前事件可恢复出的 AG 内部时间

会话标识补充：

- `session.id`
  - bridge 内部 session id
  - 由 bridge runtime 自己生成
  - 用于 `/sessions/:id/*`、持久化目录、CLI 默认路由
- `cascadeId`
  - AG 原生会话 id
  - 来自 AG / Language Server
  - 用于 `/ag-sessions/:cascadeId/*`
- `trajectoryId`
  - AG 轨迹标识
  - 主要出现在 `GET /ag-sessions` 的摘要里
  - 不是 bridge session id

## 一、VS Code 扩展命令

安装扩展后，可以在命令面板使用：

- `AG Bridge: 启动服务`
- `AG Bridge: 停止服务`
- `AG Bridge: 查看状态`
- `AG Bridge: 打开状态面板`

说明：

- `查看状态` 和 `打开状态面板` 都会进入插件内的状态页
- 状态页会显示实际监听端口、数据目录、session 列表、以及最近一次启动错误
- 状态页支持一键复制当前 `base URL`
- 状态页支持一键执行“同步 AG Sessions”

## 二、HTTP 接口

### `GET /health`

返回基础健康状态。

返回示例：

```json
{
  "ok": true,
  "sessionCount": 1,
  "liveSessionCount": 0,
  "persistedSessionCount": 1,
  "clientCount": 0,
  "defaultMode": "connect",
  "dataDir": "/Users/you/.ag-bridge"
}
```

### `GET /status`

返回服务级状态，推荐优先使用这个接口。

返回示例：

```json
{
  "ok": true,
  "running": true,
  "host": "127.0.0.1",
  "requestedPort": 9464,
  "actualPort": 9465,
  "address": "http://127.0.0.1:9465",
  "switchedPort": true,
  "dataDir": "/Users/you/.ag-bridge",
  "startedAt": "2026-03-28T10:21:42.378Z",
  "uptimeSeconds": 5,
  "sessionCount": 2,
  "liveSessionCount": 0,
  "persistedSessionCount": 2,
  "clientCount": 0,
  "defaultMode": "connect"
}
```

字段说明：

- `requestedPort`
  - 配置里请求的端口
- `actualPort`
  - 实际监听端口
- `switchedPort`
  - 是否因为端口冲突而自动切换
- `dataDir`
  - 持久化目录

### `GET /sessions`

列出所有 session。

注意：

- 当前进程内还活着的 session 会标记为 `live: true`
- 从磁盘恢复出来的历史 session 会标记为 `live: false`

### `GET /ag-sessions`

列出当前 Antigravity Language Server 里已经存在的 session。

这个接口返回的是“AG 原生会话摘要”，不是 bridge 内部 session。

查询参数：

- `workspacePath`
  - 可选
  - 只返回与指定工作区相关的 AG session

返回示例：

```json
{
  "sessions": [
    {
      "cascadeId": "csc_123",
      "trajectoryId": "traj_456",
      "summary": "请帮我总结这个仓库",
      "runStatus": "idle",
      "stepCount": 18,
      "lastUserInputStepIndex": 12,
      "workspacePaths": ["/Users/weinan/Codes/ag-bridge"],
      "attachedSessionId": "sess_789",
      "live": true
    }
  ]
}
```

字段说明：

- `cascadeId`
  - AG 原生 session 标识
- `attachedSessionId`
  - 如果已经被 bridge attach，这里会返回对应的 bridge session id
- `live`
  - 是否已经被当前 bridge 进程 attach 并实时监听

### `GET /models`

列出当前连接到的 Antigravity Language Server 返回的可用模型。

查询参数：

- `workspacePath`
  - 可选
  - 用于优先连接某个工作区关联的 LS

返回示例：

```json
{
  "models": [
    {
      "key": "Claude_4_Sonnet",
      "label": "Claude 4 Sonnet",
      "kind": "model",
      "name": "CLAUDE_4_SONNET",
      "id": 281,
      "isPremium": true,
      "isRecommended": true,
      "disabled": false
    },
    {
      "key": "Auto",
      "label": "Auto",
      "kind": "alias",
      "name": "AUTO",
      "id": 7,
      "isPremium": false,
      "isRecommended": false,
      "disabled": false
    }
  ]
}
```

说明：

- 这里返回的是“当前 LS / 当前账号环境实际给出的可用模型配置”
- 比静态源码枚举更贴近真实可用情况
- `kind=model` 表示具体模型，`kind=alias` 表示模型别名，例如 `AUTO`、`RECOMMENDED`
- 常见字段含义：
  - `key`
    - 便于脚本展示的稳定键
  - `label`
    - LS 返回的展示名
  - `name`
    - 真实模型名或别名名
  - `id`
    - 对应枚举 id
  - `isPremium`
    - 是否属于高级模型
  - `isRecommended`
    - 是否被 LS 标记为推荐模型
  - `disabled`
    - 当前是否不可用

### `POST /ag-sessions/sync`

同步当前所有 AG session。

这个接口会做两件事：

- 先发现当前 LS 中的所有 AG session
- 再把尚未 attach 的 session 全部 attach 到 bridge

请求体：

```json
{
  "workspacePath": "/Users/weinan/Codes/ag-bridge"
}
```

`workspacePath` 可选。

返回示例：

```json
{
  "discovered": [
    {
      "cascadeId": "csc_123",
      "summary": "请帮我总结这个仓库",
      "live": true
    }
  ],
  "attached": [
    {
      "id": "sess_789",
      "cascadeId": "csc_123",
      "live": true
    }
  ]
}
```

适用场景：

- 你已经在 AG UI 里手工有很多会话，想一次性把它们纳入 bridge
- 你想定时调用这个接口，持续发现“新出现的 AG 会话”

### `POST /ag-sessions/:cascadeId/attach`

把一个指定的 AG session attach 到 bridge。

请求体：

```json
{
  "workspacePath": "/Users/weinan/Codes/ag-bridge"
}
```

返回值中的 `session.id` 就是后续继续调用 `/sessions/:id/*` 和 `/sessions/:id/stream` 的 bridge session id。

如果这个 `cascadeId` 之前已经被 bridge attach 过，通常会复用已有的 bridge session id；如果没有，则会新建一个 bridge session id。

### `POST /ag-sessions/:cascadeId/messages`

直接按 AG 的 `cascadeId` 发消息。

这个接口会先自动 attach 对应 AG session，再发送消息。

请求体：

```json
{
  "text": "继续上一个问题，给我一个简短结论",
  "model": "auto"
}
```

如果你不想先单独调用 `attach`，这个接口会更方便。

### `POST /sessions`

创建新 session。

请求体：

```json
{
  "mode": "connect",
  "workspacePath": "/Users/weinan/Codes/ag-bridge",
  "model": "claude-4-sonnet",
  "sessionId": "demo_session_001"
}
```

字段：

- `mode`
  - 可选
  - `connect` 或 `launch`
  - 默认 `connect`
- `workspacePath`
  - 可选
  - 指定目标工作区
- `model`
  - 可选
  - session 默认模型
  - 支持类似 `claude-4-sonnet`、`google-gemini-2-5-pro`、`auto`、`recommended` 这类写法
- `sessionId`
  - 可选
  - 自定义 bridge session id
  - 如果不传，bridge 会自动生成

当前实现里，`session.id` 仍然是 bridge 这一层自己的标识；真正由 AG 提供的官方 id 是 `cascadeId`。

返回示例：

```json
{
  "session": {
    "id": "sess_123",
    "cascadeId": "cascade_456",
    "mode": "connect",
    "workspacePath": "/Users/weinan/Codes/ag-bridge",
    "requestedModel": "model:CLAUDE_4_SONNET",
    "live": true,
    "createdAt": "2026-03-28T07:30:00.000Z",
    "updatedAt": "2026-03-28T07:30:00.000Z",
    "bridgeCreatedAt": "2026-03-30T02:10:00.000Z",
    "bridgeUpdatedAt": "2026-03-30T02:10:00.000Z",
    "sourceCreatedAt": "2026-03-28T07:30:00.000Z",
    "sourceUpdatedAt": "2026-03-28T07:30:00.000Z",
    "runStatus": "idle",
    "latestText": "",
    "latestThinking": "",
    "eventCount": 1,
    "messageCount": 0,
    "stepCount": 0,
    "pendingApprovals": [],
    "lastEventType": "session.created"
  }
}
```

### `GET /sessions/:id`

获取单个 session 快照。

### `POST /sessions/:id/resume`

恢复一个已有 session。

说明：

- 如果该 session 当前已经是 live，会直接返回当前 live session
- 如果该 session 现在只有磁盘快照，bridge 会读取它保存的 `cascadeId` 和 `workspacePath` 并重新 attach
- 恢复成功后，通常会继续使用原来的 bridge `session.id`

### `DELETE /sessions/:id`

删除一个已经持久化但当前不再 live 的 session。

说明：

- 只允许删除非 live session
- 删除后会移除其 `snapshot.json` 和 `events.jsonl`

### `GET /sessions/:id/events`

获取 session 已记录事件。

查询参数：

- `since`
  - 可选
  - 只返回 `seq > since` 的事件
- `limit`
  - 可选
  - 只返回最后 `limit` 条事件

返回示例：

```json
{
  "session": {
    "id": "sess_123",
    "live": false
  },
  "events": [
    {
      "seq": 12,
      "type": "cascade.text.delta"
    }
  ]
}
```

### `GET /sessions/:id/export`

导出单个 session 的完整快照和事件列表。

这个接口适合做：

- 调试导出
- 持久化备份
- 二次分析
- 给外部脚本做结构化消费

### `POST /sessions/:id/messages`

给指定 live session 发送消息。

请求体：

```json
{
  "text": "请总结当前仓库",
  "model": "google-gemini-2-5-pro"
}
```

说明：

- `model` 可选
- 如果不传，就使用 session 创建时记录的默认模型
- 如果传了，会只覆盖当前这一条消息

### `POST /sessions/:id/cancel`

取消指定 live session 当前执行。

### `POST /sessions/:id/approvals/:stepIndex/approve`

批准某个待审批步骤。

请求体：

```json
{
  "scope": "once"
}
```

`scope` 可选值：

- `once`
- `conversation`

## 三、SSE 接口

### `GET /sessions/:id/stream`

订阅 live session 的实时事件流。

查询参数：

- `since`
  - 可选
  - 先补发 `seq > since` 的历史事件，再继续实时推送

说明：

- 这个接口只针对 live session
- 如果 session 已经只是磁盘记录，不再 live，那么请使用 `GET /sessions/:id/events`

### SSE 事件名

- `ready`
- `event`

### `ready` 示例

```json
{
  "session": {
    "id": "sess_123",
    "cascadeId": "cascade_456",
    "live": true
  }
}
```

### `event` 示例

```json
{
  "seq": 12,
  "sessionId": "sess_123",
  "cascadeId": "cascade_456",
  "timestamp": "2026-03-28T07:31:00.000Z",
  "recordedAt": "2026-03-30T02:12:01.000Z",
  "sourceTimestamp": "2026-03-28T07:31:00.000Z",
  "type": "cascade.text.delta",
  "data": {
    "stepIndex": 3,
    "delta": "OK",
    "fullText": "OK",
    "sourceTimestamp": "2026-03-28T07:31:00.000Z",
    "latestTimestamp": "2026-03-28T07:31:00.000Z"
  }
}
```

## 四、Session 快照字段

`SessionSnapshot` 当前常用字段如下：

- `id`
- `cascadeId`
- `mode`
- `workspacePath`
- `live`
- `createdAt`
- `updatedAt`
- `bridgeCreatedAt`
- `bridgeUpdatedAt`
- `sourceCreatedAt`
- `sourceUpdatedAt`
- `runStatus`
- `latestText`
- `latestThinking`
- `eventCount`
- `messageCount`
- `stepCount`
- `pendingApprovals`
- `lastEventType`
- `lastError`

补充说明：

- `live`
  - `true` 表示当前进程里还能继续收发
  - `false` 表示只是从磁盘恢复的历史记录
- `messageCount`
  - 已发送的用户消息数
- `stepCount`
  - 当前已观察到的步骤数
- `createdAt` / `updatedAt`
  - 对外主时间
  - 优先使用 AG 内部时间；没有时回退为 bridge 时间
- `bridgeCreatedAt` / `bridgeUpdatedAt`
  - bridge 本地创建 / 最后记录时间
- `sourceCreatedAt` / `sourceUpdatedAt`
  - 已恢复出的 AG 内部创建 / 最后更新时间

## 五、事件类型

当前已经稳定对外暴露的事件类型包括：

- `session.created`
- `message.sent`
- `session.cancel.requested`
- `cascade.user_input`
- `cascade.status`
- `cascade.step.new`
- `cascade.step.updated`
- `cascade.text.delta`
- `cascade.thinking.delta`
- `cascade.command.output`
- `cascade.approval.needed`
- `cascade.approval.resolved`
- `cascade.done`
- `cascade.error`
- `cascade.fallback.polling`

补充说明：

- `cascade.step.new` / `cascade.step.updated` 的 `data.step` 现在会额外带：
  - `sourceTimestamp`
  - `latestTimestamp`
  - `timing`
- `data.step.timing` 会展开 AG metadata 中已知的时间字段：
  - `chatStartCreatedAt`
  - `createdAt`
  - `viewableAt`
  - `finishedGeneratingAt`
  - `lastCompletedChunkAt`
  - `completedAt`
  - `latestStatusTransitionAt`
  - `statusTransitions`

其中：

- `cascade.step.new` / `cascade.step.updated`
  - 这两个事件除了保留顶层摘要字段，还会在 `data.step` 下带完整 step 详情
  - `data.step.raw` 是 protobuf step 的完整 JSON 展开，默认值也会带出来
  - `data.step.payload` 是当前 oneof step payload 的完整详情，适合前端按 step 类型直接消费
- `cascade.user_input`
  - 代表该 AG session 中出现了一条用户输入
  - 不管这条输入是脚本通过 bridge 发出的，还是你在 AG 对话框里手工输入的，只要该 session 已经 attach，bridge 都会把它记成事件

## 六、如何同步 AG 里手工会话

如果你希望脚本不仅能控制 bridge 创建的会话，也能接收你在 AG UI 里手工输入的内容，推荐按这个流程：

1. 先从插件状态页复制当前 `base URL`
2. 调用 `POST /ag-sessions/sync`
3. 从返回结果里拿到每个 `attached[].id`
4. 对这些 `session id` 订阅 `GET /sessions/:id/stream`
5. 之后：
   - 你在 AG 对话框里手工输入，会收到 `cascade.user_input`
   - AG 回复时，会收到 `cascade.text.delta`
   - AG 的 thinking、命令输出、审批请求，也会继续走原有事件流
6. 如果后来你又新开了一个 AG 会话，再调用一次 `POST /ag-sessions/sync`

一个最小示例：

```bash
curl -X POST http://127.0.0.1:9464/ag-sessions/sync \
  -H 'Content-Type: application/json' \
  -d '{}'
```

然后订阅：

```bash
curl -N http://127.0.0.1:9464/sessions/<sessionId>/stream
```

## 七、Python SDK / CLI

Python 侧没有重写 bridge 核心，而是直接调用这套本地 HTTP 接口。

目录：

- `python/`

本地安装：

```bash
pip install -e python
```

CLI 示例：

```bash
ag-bridge-py status
ag-bridge-py model-list
ag-bridge-py session-list
ag-bridge-py session-create --session-id demo_session_001
ag-bridge-py ask "请只回复 OK" --thinking off
ag-bridge-py ask "继续这个会话" --session demo_session_001
ag-bridge-py resume --last
ag-bridge-py chat --last
ag-bridge-py stream demo_session_001 --thinking off
```

说明：

- Python CLI 对齐了 Node CLI 的核心 session 管理能力
- 支持 `resume`、`chat`、`stream`、`delete`
- `ask` 支持 `--session`、`--last`、`--session-id`、`--thinking`
- 当前它和 Node CLI 一样，都是仓库内附带的本地工具，不走单独的发布流程

Python 示例：

```python
from ag_bridge_client import BridgeClient

client = BridgeClient("http://127.0.0.1:9465")
models = client.list_models()
print(models)

ag_sessions = client.list_ag_sessions()
print(ag_sessions)

synced = client.sync_ag_sessions()
print(synced["attached"])

result = client.ask("请只回复 OK")
print(result["text"])
```
