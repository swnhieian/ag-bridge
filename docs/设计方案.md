# 设计方案

## 目标

这一版的目标已经从“把 AG 接出来”进一步收敛成四件事：

- 以 `antigravity-client` 为底层，保留稳定的 LS 路线
- 做成真正可用的 VS Code 扩展方案，而不是只在终端里勉强可跑
- 把 session 和事件持久化，支持服务重启后的恢复
- 同时提供适合自动化集成的 Python SDK，而不是在 Python 里重写 bridge 核心

## 为什么底层继续选 `antigravity-client`

和 DOM 抓取、CDP 点按钮相比，`antigravity-client` 的核心优势仍然没有变：

- 直接面向 Antigravity Language Server
- 能 `connect()` 已运行实例，也能 `launch()` 新实例
- 能拿到 cascade / history / approval / command output
- 不依赖前端按钮、文案、CSS 选择器

也就是说，这条链路虽然仍然不是官方公开 API，但相比 UI 自动化已经稳定很多。

## 当前整体架构

```text
VS Code 扩展宿主
    └─ ExtensionBridgeController
         ├─ BridgeHttpServer
         │    ├─ BridgeRuntime
         │    │    ├─ BridgeSession
         │    │    └─ BridgePersistenceStore
         │    └─ HTTP / SSE
         └─ 状态面板（Webview）

外部脚本 / JS CLI / Python SDK
    └─ 本地 HTTP / SSE
         └─ AG Bridge
```

## 组件职责

### 1. `BridgeSession`

负责单个 cascade：

- 发送消息
- 取消执行
- 跟踪 `latestText / latestThinking / runStatus`
- 跟踪审批请求
- 产生标准化事件流

### 2. `BridgePersistenceStore`

负责磁盘持久化：

- `snapshot.json`
- `events.jsonl`
- 重启后的历史 session 恢复

当前目录结构：

```text
dataDir/
  meta.json
  sessions/
    <sessionId>/
      snapshot.json
      events.jsonl
```

### 3. `BridgeRuntime`

负责：

- 管理底层 `AntigravityClient`
- 复用相同工作区 / 模式的 client
- 合并 live session 与磁盘恢复 session
- 对外提供导出、删除、事件读取等能力

### 4. `BridgeHttpServer`

负责：

- 本地 HTTP / SSE 服务
- 端口冲突时自动顺延
- `status / sessions / export / stream` 等接口

### 5. VS Code 扩展层

负责：

- 在扩展宿主内启动 bridge
- 提供命令面板入口
- 提供状态栏入口
- 提供 Webview 状态页

## 稳定性策略

### 1. 继续坚持 LS 优先

主链路仍然只依赖：

- LS 发现
- Connect RPC
- trajectory / history

不依赖：

- DOM
- webview 结构
- 按钮文本
- CDP 页面操作

### 2. Reactive 不可用时自动降级

这台机器上的真实环境里，LS 返回过：

- `reactive state is disabled`

因此 bridge 内部已经实现自动降级：

- 优先尝试 reactive stream
- 失败后自动切到 `getHistory()` 轮询
- 通过 diff 继续产出：
  - `cascade.text.delta`
  - `cascade.thinking.delta`
  - `cascade.command.output`
  - `cascade.step.new`
  - `cascade.step.updated`
  - `cascade.approval.needed`
  - `cascade.done`

这意味着即使 reactive 被禁用，系统依然能工作。

### 3. 插件内直接展示真实状态

以前的“状态”更像一个提示框，现在变成了真正的状态面板。

主要展示：

- 期望端口 / 实际端口
- 服务地址
- 数据目录
- 工作区
- session 总量与 live 数
- session 表格
- 最近一次启动错误

## 为什么不在 Python 里重写 bridge 核心

这是这次设计里非常重要的一点。

可选路线其实有两种：

### 路线 A：Python 重写核心桥

优点：

- 最终集成端是 Python

问题：

- 要重新处理 LS 连接、事件流、审批、fallback、会话模型
- 维护两套 bridge 核心
- 稳定性和演进成本都会变差

### 路线 B：保留 TS 核心，Python 只做客户端

优点：

- 只有一套 bridge 核心
- VS Code 扩展仍然是主形态
- Python 集成依旧很方便
- 外部系统只要调用本地 HTTP 接口即可

因此当前选择的是：

- TS / VS Code 扩展 = 核心桥
- Python = 零依赖 SDK / CLI

这是更稳、更适合长期维护的方案。

## CLI 设计方向

CLI 现在分成两层：

### 1. 结构化命令

适合脚本和自动化：

- `status`
- `session:create`
- `session:list`
- `events`
- `export`
- `approve`
- `delete`

### 2. 交互式 REPL

适合人工调试和半自动使用：

- `chat`
- `/sessions`
- `/use`
- `/events`
- `/approve`
- `/thinking on|off`

并且 CLI 现在支持：

- `--base-url`

这样即使默认端口被占用、服务自动切到别的端口，也能继续方便地接入。

## 当前实现范围

已经实现：

- session 创建 / 消息发送 / 取消 / 审批
- HTTP / SSE
- 持久化存储与重启恢复
- session 导出与删除
- 端口冲突自动切换
- JS CLI
- 交互式 CLI
- Python SDK / CLI
- VS Code 状态面板

还没做：

- artifact 专用接口
- `launch` 模式完整回归
- 更细的多工作区选择 UI
- 更强的富文本终端展示

## 推荐使用方式

最推荐的用法现在是：

1. 以 VS Code 扩展形态运行 `AG Bridge`
2. 通过插件内状态面板观察端口和 session
3. 外部脚本通过：
   - JS CLI
   - Python SDK
   - 直接 HTTP / SSE

这也是目前稳定性、可维护性和集成便利性之间最平衡的方案。
